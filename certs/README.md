# PKI и конфигурация Kubernetes (The Hard Way)

Этот этап посвящен созданию инфраструктуры открытых ключей (PKI) и генерации файлов конфигурации (kubeconfig), которые связывают сертификаты и адреса серверов в единые пакеты для компонентов кластера.

## 1. Генерация сертификатов (generate_certs.sh)

Kubernetes использует сертификаты для аутентификации компонентов и шифрования трафика. Мы используем утилиту cfssl для создания собственного Центра Сертификации (CA) и выпуска всех необходимых сертификатов.

### Зачем нужны эти сертификаты?
 * CA (Certificate Authority): Корень доверия. Все компоненты кластера доверяют сертификатам, подписанным этим CA.
 * Admin: Для административного доступа (пользователь admin, группа system:masters). Группа system:masters дает полные права в кластере через RBAC.
 * Kubelet (Workers): Каждая нода (worker-0, worker-1 и т.д.) получает свой сертификат. Это нужно для Node Authorizer, чтобы воркеры могли запрашивать только свои данные.
 * Controller Manager / Scheduler / Proxy: Служебные компоненты, использующие сертификаты для безопасного подключения к API Server.
 * Kubernetes API Server: Самый сложный сертификат. Он должен содержать все возможные IP и DNS-имена, по которым к нему могут обратиться (включая локальные адреса, IP контроллеров и виртуальный IP сервиса Kubernetes).
 * Service Account: Пара ключей для подписи и проверки токенов (JWT), которые выдаются подам.
 * ETCD: Отдельная цепочка доверия для базы данных кластера. Мы генерируем уникальные сертификаты для каждого узла etcd (для peer-to-peer общения) и клиентский сертификат для API Server, чтобы он мог читать/писать в базу.

### Как запустить:

```bash
chmod +x generate_certs.sh
./generate_certs.sh
```

## 2. Генерация Kubeconfigs (generate_configs.sh)

kubeconfig — это файл, который объединяет адрес API-сервера, корневой сертификат CA и клиентский сертификат/ключ в один документ. Это избавляет от необходимости указывать все параметры в командной строке.

### Что мы генерируем:
 * Worker Configs: Для каждого воркера создается свой конфиг, чтобы они могли стучаться в API Server.
 * Kube-Proxy Config: Общий конфиг для сетевого прокси на всех узлах.
 * Controller Manager & Scheduler: Конфиги для локальных сервисов управления на контроллерах.
 * Admin Config: Ваш "пропуск" для управления кластером через kubectl.
 * ETCD Configs: Специфичные конфиги для взаимодействия с базой данных на каждом контроллере.

### Важные параметры:
 * KUBERNETES_ADDRESS: В скрипте захардкожен IP 10.0.0.10 (первый контроллер). В продакшене здесь обычно стоит IP балансировщика.
 * --embed-certs=true: Мы упаковываем сертификаты прямо внутрь конфига. Это делает файл самодостаточным и удобным для копирования на удаленные ноды.

### Как запустить:

```bash
chmod +x generate_configs.sh
./generate_configs.sh
```

## Итоговая структура файлов
После запуска обоих скриптов у вас появится набор файлов:

* `*.pem` и `*-key.pem` — сертификаты и закрытые ключи.
* `*.kubeconfig` — готовые конфиги для компонентов.

> **Безопасность**: Файлы с суффиксом `-key.pem` являются секретными. Не передавайте их по незащищенным каналам.

## Что дальше?
Полученные файлы необходимо распределить по узлам:

1. **Контроллеры**: Нуждаются в CA, сертификатах API, etcd, сервисных аккаунтов и соответствующих конфигах.
2. **Воркеры**: Нуждаются в CA, своих сертификатах и своих `.kubeconfig`.

Это удобно делать через Ansible, используя сгенерированный Terraform инвентарь.

### Почему это сделано именно так?
 1. Локация: В сертификатах указан Minsk, Belarus, что соответствует вашему физическому окружению.
 2. Разделение прав: Мы не используем один "общий" сертификат. У каждого компонента свои права (RBAC), что минимизирует ущерб при компрометации одной ноды.
 3. Автоматизация: Скрипты используют массивы CONTROLLERS и WORKERS, соответствующие IP-адресам из terraform.tfvars. Это гарантирует, что сертификаты будут валидны для созданных виртуальных машин.
