# K8s The Hard Way: Окружение на QEMU/KVM + Terraform

Этот проект разворачивает виртуальную инфраструктуру для прохождения "Kubernetes The Hard Way" локально с использованием гипервизора QEMU/KVM и Terraform. 

Конфигурация автоматически создает 3 control-plane узла и 3 worker-узла, настраивает им статические IP-адреса и генерирует инвентарный файл для последующей настройки через Ansible.

## Требования

Для работы инфраструктуры необходимо иметь:
* **QEMU/KVM и Libvirt**: Установленные и запущенные.
* **Terraform**: Версии >= 0.13.
* **Minio / S3-совместимое хранилище**: Запущенное локально (в данном примере на `http://localhost:9000`) для хранения `.tfstate` файла.
* **Ubuntu Cloud Image**: Скачанный образ (по умолчанию ожидается по пути `/var/lib/libvirt/images/noble-server-cloudimg-amd64.img`).
* **SSH ключи**: Сгенерированная пара ключей Ed25519 в `~/.ssh/`.

## Работа с секретами (S3 Backend)

State-файл Terraform хранится в Minio. Чтобы не коммитить учетные данные от хранилища в открытом виде (в файлах `providers.tf` или `variables.tf`), мы используем переменные окружения AWS.

### Шаг 1. Создание локального окружения
Создайте файл `.env.local` в корне папки с Terraform (убедитесь, что он добавлен в `.gitignore`!):

```bash
cat <<EOF > .env.local
export AWS_ACCESS_KEY_ID="admin"
export AWS_SECRET_ACCESS_KEY="password"
EOF
```

### Шаг 2. Загрузка переменных
Перед выполнением любых команд Terraform, загрузите переменные в текущую сессию терминала:

```bash
source .env.local
```

## Использование
**1. Инициализация Terraform**
Инициализирует провайдер libvirt и настраивает подключение к S3-бэкенду:

```bash
terraform init
```

**2. Проверка плана выполнения**
Покажет, какие ресурсы (виртуальные машины, диски, сеть) будут созданы:

```bash
terraform plan
```

**3. Развертывание инфраструктуры**
Применение конфигурации:

```bash
terraform apply
```

**4. Уничтожение инфраструктуры**
Когда кластер больше не нужен:

```bash
terraform destroy
```

## Что происходит под капотом?
При запуске terraform apply выполняется следующий сценарий:
 1. **Создание сети (network.tf)**: Создается NAT-сеть k8s-net (10.0.0.0/24) в libvirt с локальным DNS.
 2. **Подготовка базового образа (main.tf)**: На основе загруженного Ubuntu Cloud Image создается базовый том (qcow2).
 3. **Cloud-Init (cloud_init.cfg)**: Генерируются ISO-образы с метаданными. В них пробрасывается ваш публичный SSH-ключ, отключается парольная аутентификация и устанавливаются необходимые системные пакеты (включая qemu-guest-agent для нормальной работы внутри консоли гипервизора).
 4. **Provisioning ВМ (main.tf)**: Разворачиваются 6 виртуальных машин. Каждая получает свой диск, подключение к сети с зафиксированным MAC/IP и cloud-init диск. Доступ к ВМ настроен только через консоль (PTY/Serial), что идеально для headless-виртуализации без зависимостей X11/Wayland.
  5. **Генерация инвентаря (main.tf, inventory.tmpl)**: После успешного создания всех ВМ, Terraform автоматически рендерит файл ansible/inventory.ini, распределяя хосты по группам [controllers] и [workers].

## Подключение к ВМ
После применения конфигурации вы можете зайти на любую ноду по SSH, используя ключи:

```bash
ssh vladimir@10.0.0.10  # Для controller-0
ssh vladimir@10.0.0.20  # Для worker-0
```

Или использовать `ansible -m ping all -i ../ansible/inventory.ini` для проверки связности перед следующим этапом настройки кластера.
